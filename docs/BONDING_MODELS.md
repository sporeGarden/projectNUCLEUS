# Bonding Models — How Gates Trust Each Other

NUCLEUS uses chemistry-inspired bonding to describe trust relationships
between gates (machines running primals). The bonding model determines
what a gate can see, what it can run, and how much it's trusted.

4 NUCs, DDR3 through DDR5, RTX 2070 through RTX 5090, 32 GB through
256 GB RAM. Each gate runs the NUCLEUS atomic that matches its hardware.
Bonding describes how these gates relate to each other and to the outside.

## Covalent Bond

**Analogy**: Shared electrons. Atoms in a molecule.
**Trust**: Full. Shared family seed via BearDog.
**Use case**: Your own machines, connected by ethernet.

Covalent gates share a family seed generated by `seed_workflow.sh init`.
Each gate gets a lineage seed derived from the family root. BearDog
authenticates peers using this shared lineage. Songbird discovers
covalent peers on the LAN automatically via BirdSong UDP multicast
(`239.255.0.1:4200`).

All capability sharing is unrestricted within a covalent bond. If
one gate has a GPU and another has 256 GB RAM, workloads route
freely between them based on what the workload needs.

**Deploy graph**: `basement_hpc_covalent.toml`

The GMKtec NucBox M6 connects to the active gate via USB-C ethernet, creating


### Covalent Mesh Target

The 10G backbone switch is acquired and NICs are installed. Once cabled,
the covalent mesh expands from single-gate to multi-gate:

```
[the active gate] ←10G→ [strandGate] ←10G→ [southGate]
     ↕                  ↕
[eastGate]         [westGate / cold storage]
```

Each gate runs its matched atomic. Songbird mesh discovery auto-registers
peers. biomeOS routes workloads to the best gate for the job:
- `compute.*` → strandGate (64 cores + 3090) or southGate (128 GB RAM)
- `storage.*` → westGate (76 TB ZFS)

**What this is**: The basement HPC. Heterogeneous hardware united by
genetic trust. The Celeron NUC runs the same primal binary as the
dual EPYC. That's the covalent principle.

## Ionic Bond

**Analogy**: Electron transfer. Oppositely charged ions.
**Trust**: Scoped. Capability tokens, not family seeds.
**Use case**: ABG compute sharing, external collaborators, friends.

Ionic bonds create a metered relationship between two families. The
guest gets a capability token granting specific permissions (e.g.,
"submit workloads of type native, max 4 GB RAM, max 1 hour"). The
host's family seed is never shared.

biomeOS tracks ionic usage via metering. The guest sees a facade —
they know their workload ran, they get results, but they don't see
the internal mesh topology, BearDog's family seed, or Songbird's
LAN discovery.

**Deploy graph**: `ionic_capability_share.toml`

### ABG — First Ionic Instance

ABG (Accelerated Bioinformatics Group) is a bioinformatics research
community (~30 members). They need compute for scRNA-seq, bulk RNA-seq,
metagenomics, and structure prediction. The ionic pattern:

```
[ABG collaborator]
      ↓ BTSP-authenticated tunnel
[NUC intake / Tower Atomic]
      ↓ Cat6e LAN (covalent internal)
[the active gate / JupyterHub + Nest Atomic]
      ↓ toadStool dispatch
[strandGate / heavy compute]
```

The collaborator submits workloads. BTSP Phase 3 encrypts the channel.
Their workload runs on internal hardware. They get results + provenance.
They never see the internal topology.

**Security evolution**: Phase 2a uses Cloudflare Tunnel (`cloudflared`)
on the NucBox M6 intake — zero NAT config, HTTPS for free, production
metrics as calibration baselines. Each subsequent step replaces one
Cloudflare component with a primal, validated via `infra/benchScale`
load and pen testing. Phase 4 is full BTSP — zero external dependencies.
See `specs/TUNNEL_EVOLUTION.md` for the systematic replacement plan.

### Sub-Goal: Sovereign JupyterHub via primals.eco

**Target**: An ABG collaborator navigates to `primals.eco/compute`,
authenticates via BTSP, and gets a JupyterHub notebook environment on
the active gate. The security chain:

```
Browser → primals.eco → BTSP tunnel (BearDog + Songbird)
       → petalTongue reverse proxy → JupyterHub on the active gate
       → notebook kernels (Python/R bioinformatics)
       → toadStool dispatch for heavy workloads
       → provenance pipeline (rhizoCrypt → loamSpine → sweetGrass)
```

What makes this ionic (not covalent): the collaborator has a scoped
capability token, not a family seed. They can run notebooks and submit
workloads but cannot discover internal gates, access raw storage, or
see the mesh topology.

**Dependencies for the sub-goal**:
1. petalTongue `web` mode serving primals.eco content (Phase 3 primal)
2. BearDog BTSP Phase 3 tunnel termination at the intake node
3. Songbird NAT traversal replacing Tailscale
4. JupyterHub proxied behind the BTSP tunnel
5. Ionic metering via biomeOS (tracking usage per capability token)

**What exists today** (from `wateringHole/compute-sharing/`):
- `setup-jupyterhub.sh` — portable JupyterHub install with Python + R kernels
- `TUNNEL_ACCESS_GUIDE.md` — Tailscale/SSH/WireGuard options
- `SOVEREIGN_COMPUTE_SHARING.md` — full pattern doc (Phase 0-4)
- JupyterHub has been running on the active gate since Phase 1 validation

## Metallic Bond

**Analogy**: Electron sea. Delocalized electrons shared across a lattice.
**Trust**: Institutional. Homogeneous compute pool.
**Use case**: University HPC, datacenter fleet, ICER cluster.

Metallic bonds describe pools of interchangeable compute nodes. Any
node can serve any workload. Capabilities are delocalized — the pool
has aggregate TFLOPS, not per-node assignments.

biomeOS load-balances across the metallic pool. Individual node
identity matters less; what matters is aggregate capacity and
availability.

**Deploy graph**: `nucleus_complete.toml` (bonding_policy: metallic)
**Example**: 50 identical rack nodes at ICER, each running Node Atomic.
biomeOS distributes wetSpring workloads across the pool based on
current load.

**projectNUCLEUS scope**: This is Phase 4 territory. The covalent
basement proves the model; metallic federation scales it to
institutional hardware. foundation (the external-facing project) owns
the relationship with institutions. projectNUCLEUS owns the deploy
graphs and composition validation that make metallic bonding work.

## Weak Forces

**Analogy**: Van der Waals forces. Transient, low-energy.
**Trust**: Minimal. Pre-authentication state.
**Use case**: Dark Forest beacons, initial contact, external API calls.

Weak bonds are the default state before authentication. A new gate
announces its presence via Dark Forest beacon. Until BearDog verifies
its lineage, it operates under weak-force rules: no workload dispatch,
no data access, read-only capability discovery.

Weak forces naturally strengthen: a verified beacon becomes ionic
(metered access), then potentially covalent (family adoption) as
trust is established.

**Example**: A new NUC appears on the network. Songbird discovers it.
BearDog has not verified it yet. It can announce capabilities but
cannot receive workloads until bonding is established.

## Mixed Bonding

Real deployments mix bonding types. The active gate topology today:

```
[Basement LAN — Covalent]
  the active gate ←10G→ strandGate ←10G→ southGate ←10G→ eastGate
       ↕                                               ↕
  westGate (cold storage)                         biomeGate (HBM2)
       ↕
[NUC Intake — Covalent to LAN, Ionic to external]
       ↕
[ABG Collaborator — Ionic / BTSP tunnel]
       ↕
[primals.eco — Weak public / petalTongue]
```

The `organo_metal_salt_complex.toml` graph in primalSpring models
exactly this: covalent core, ionic edges, weak public Songbird.
Deploy graphs encode bonding policy in `[graph.bonding_policy]`:

```toml
[graph.bonding_policy]
tower_internal = "covalent"       # LAN gates trust fully
tower_to_nucleus = "metallic"     # institutional compute pools
cross_family = "ionic"            # ABG collaborators
encryption_tiers.tower = "full"   # BTSP Phase 3 AEAD
encryption_tiers.cross_family = "full"
encryption_tiers.nucleus = "hmac_plain"
```

## Scope Boundary: projectNUCLEUS vs foundation

**projectNUCLEUS** focuses inward: execution, composability, and
invisibility of primals on **the active gate**. It owns the covalent basement
HPC and the ionic ABG interactions. The ABG deployment is the primary
validation vehicle — every ABG workload that succeeds proves that
primalSpring's upstream patterns (deploy graphs, BTSP, discovery,
provenance) work in production.

**foundation** focuses outward: institutional relationships, metallic
federation with universities, public documentation, the case for
sovereign compute. foundation builds on what projectNUCLEUS proves.

The boundary: projectNUCLEUS deploys and validates primalSpring patterns
on real hardware under real external load. foundation takes that proof
and makes the case to institutions.
